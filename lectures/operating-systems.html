Operating Systems

Accounting Architecture

Software is a fundamental part of technology. Software performs all of the automated tasks and facilitates the manual tasks in the information system. Software stores and retrieves data; software analyses and generates reports. Without software, none of the other technology components could serve their intended purpose. The operating system (OS) is a fundamental piece of software. It provides a platform for the functionality of the information system.

Nothing ‘Just Works’

Users of personal computers and mobile devices frequently report that their motivation for selecting a particular operating system or platform was that it “just works.” This is sufficient for an end user because the purpose of the operating system is to facilitate the functionality of the information system by getting out of the user's way. The user should be as little aware of the operating system as possible, and in many situations, especially with Web content, the user never interacts with the operating system at all. An accounting architect, on the other hand, must direct the operating system to meet the users' needs. As a result, the ability to participate in systems design and maintenance relies in large part on familiarity with enterprise-grade operating systems.

OSS OS

The capabilities of the operating system have a strong influence over of the capabilities of the information system, and open source has had a strong influence over the capabilities of enterprise-grade operating systems. Despite the popularity of Microsoft Windows and Apple macOS on personal computers, the most widely used enterprise-grade operating systems are open source. Furthermore, the open source development of enterprise-grade operating systems has galvanized the open source movement, and the wide adoption of open source software in enterprise-grade information systems is due to the successes in developing open source operating systems.

Unintended Consequences

The original creator of the most successful enterprise-grade OS had not intended to develop the software for enterprise use. This is one of the marks of crowdsourcing and open sourcing: the influence of the community at large over the final product. Eric Raymond illustrates this effect in his book The Cathedral and the Bazaar. The cathedral represents a development model, in which the creator determines the direction of the product. The bazaar represents a development model, in which the community jointly determines the direction of the product. (Many understand this to be a comparison between closed and open source development, but it is actually a comparison of two open source products.) Because of the open mentality, the community grew this OS project to be much larger than the creator had envisioned.

UNIX

The history of current enterprise-grade operating systems begins with UNIX. Although it is not an open source OS, its development model, as well as its history, have strongly influenced the functionality its open source competitors.

In 1969, researchers at AT&T Bell Laboratories became dissatisfied with Multics, the operating system that they had begun to develop with MIT and General Electric, and they decided to develop an alternative operating system, which they named UNIX. UNIX engineers operated under a philosophy that continues to govern software development: Write programs that do one thing and do it well. One popular representation of this philosophy is KISS (Keep It Simple, Stupid!).

Licensing

In 1973, AT&T began selling UNIX licenses—AT&T was a regulated company and could only sell licenses and not software—to universities and corporations. Unlike common current software licenses, UNIX licenses included source code and no other support. The availability of source code and the lack of commercial support motivated licensees to form communities that supported, modified, and improved on the operating system. Several of these development groups were founded by universities and funded by research grants, which was common for software development at that time. These efforts were similar to open source development, except that users could only share their improvements with other UNIX licensees because no one without a license was allowed to own a copy of UNIX.

Microsoft Who?

As an aside, in 1978, a small, unknown company named Microsoft purchased a UNIX license and began creating the UNIX-based operating system Xenix. UNIX was designed for large servers or mainframes, and Bill Gates designed Xenix for what he believed was the impending proliferation of personal computers. Microsoft eventually abandoned Xenix in favor of MS-DOS, which was the foundation for the most popular operating system for personal computers: Microsoft Windows. However, the brief existence of Xenix is evidence of a very important point: UNIX is not merely alternatives to Microsoft Windows. UNIX was the operating system of choice before anyone had heard of Microsoft. The subsequent creation of UNIX-like operating systems—operating systems that mimic the functionality of UNIX—reinforces its importance.

System V

In 1983, AT&T struck a blow against crowdsourcing. Bell Systems broke up after an antitrust case, and AT&T became able to sell UNIX software, not only licenses. AT&T released a commercialized UNIX System V and forbade access to UNIX source code. This restriction essentially ended community UNIX development.

Knowing no other model than secrecy for collecting profits from software and no other model than centralized control for developing a commercial product, AT&T clamped down hard on source-code distribution.
– Eric Raymond

BSD

Before AT&T stopped distributing UNIX source code, the University of California, Berkeley had purchased a UNIX license and begun modifying the operating system. Berkeley subsequently released their modifications as a separate operating system, which they called BSD (Berkeley Software Distribution). Through the fourth release of BSD (4BSD), users needed to have purchased a UNIX license from AT&T. Interestingly, many licensees, including employees of AT&T's Bell Labs, preferred 4BSD to System V, and the Defense Advanced Research Project (DARPA) funded the creation of the Computer Systems Research Group (CSRG) at Berkeley to develop a BSD for ARPAnet, the precursor to the Internet.

OS Importance

DARPA's decision to fund Berkeley's CSRG highlights the importance of the operating system's capabilities for facilitating an information system. When DARPA approached the CSRG, no operating system existed that would have supported the ARPAnet experiment. DARPA had reason to believe that BSD UNIX would be the best candidate to provide the necessary functionality.

Selecting the right operating system may have been more crucial then because operating systems differed more significantly, but today, companies continue to make similar evaluations. The majority have concluded that open source provides the best operating system for their specific information system needs.

An Open Standard

Despite the appeal of both System V Unix and BSD Unix, disputes arose between the programming communities. In an attempt to reconcile differences, the Institute of Electrical and Electronics Engineers (IEEE) developed a set of Portable Operating System Interface (POSIX) standards based on System V UNIX which continue to govern UNIX and UNIX-like operating systems today. The list of POSIX-certified operating systems currently contains the three most prominent Unix offerings: IBM AIX, HP-UX and Oracle Solaris. Although BSD is not POSIX-certified, it is mostly POSIX compliant. Today, the POSIX standard continues to be a valuable standard for enterprise-grade open source operating systems.

BSD Becomes Free

Starting in 1992, CSRG began replacing the remaining AT&T code in 4BSD with code that they had written. Once CSRG had removed all AT&T code, it released 386BSD as a free Unix-compatible operating systems for Intel's 80386 processor, an early iteration of the x86 line of CPUs. This was the first prominent example of an operational, enterprise-grade, open source operating system. Similar to the early days of UNIX, communities immediately formed to continue development of 386BSD. This lead to two forks of 386BSD: NetBSD for technical users and FreeBSD for less technical personal computer users. Because the CSRG released 386BSD freely to users without UNIX licenses, AT&T sued Berkeley claiming that 386BSD contained proprietary code.

After the Suit

Berkeley was finally exonerated, and the court only required minor changes to the BSD source code. The resulting code was released as 4.4BSD-lite. Unfortunately, the legal uncertainty during the two-year suit encouraged many BSD community members and developers to transition to a new, alternative open source, UNIX-like operating system called Linux. However, BSD remains widely used and supported, and many flavors (i.e., operatings systems based on BSD) exist. The most prominent are FreeBSD, NetBSD, OpenBSD (focused on security) and macOS. Furthermore, most operating systems, including Microsoft Windows, use network tools from BSD. In 2006, InformationWeek rated 4BSD as the “Greatest Software Ever Written” and the “Theoretical undergirder of the Internet.”

Linux

Linux is currently the most widely adopted operating system. It began in 1991 when Linus Torvalds, a Finnish university student, decided to develop a free UNIX-compatible kernel from scratch as a hobby. Today, 70% of internet servers, 55% of mobile devices and 97% of supercomputers run Linux. It is also the operating system of choice for many single-board computers, such as the Raspberry Pi. Although the Linux is one development project, the resulting operating system has over 600 different versions (called distributions or distros, for short). The developers of each of these distributions benefit from the open source nature of Linux by being able to modify the kernel and userland to provide a specialized user experience.

Operating System

Operating systems include a kernel and userland. They also rely on two additional supporting tools: a firmware interface and a bootloader. In order to keep the operating system out of the user's way, many operating systems package the kernel, userland, and bootloader together. Linux and BSD are modular. They each only have one kernel, but they offer multiple different userland environments and bootloaders. Modularity is a key characteristic of an open system. It allows the users to create different systems by choosing different combinations of tools.

At the Heart of the Matter

The kernel is the part of the operating system that interacts with the computer hardware. This is the lowest level of the operating system. The kernel receives requests from user applications to access computer hardware, such as the CPU, RAM, storage, audio, or video, and forwards these requests to the hardware components. Once the request is fulfilled, the kernel returns the results of the request to the application. For example, if the user wishes to listen to an audio file, the application sends the audio to the kernel, and the kernel forwards the audio to the motherboard and the speakers.

When the kernel is not familiar with a certain type of hardware, users can add device drivers as kernel modules to provide the kernel with additional functionality.

Services

User applications communicate with the kernel by means of services. Services receive and process requests, manage access to centralized resources, and deliver data. They can be part of the kernel (monolithic kernel), or they can be in userland and communicate with the kernel (microkernel). Both Linux and BSD have monolithic kernels. MacOS and iOS use a hybrid kernel, which includes some services in the kernel and some in userland.

The term “service” is motivation for the word server. Although “server” is frequently used to identify a specific type of computer hardware, a more appropriate definition would be software that provides a service. Several services run as daemons which are applications that run in the background without user input.

Userland

Userland is the part of the operating system that user interacts with. The user interacts with userland by means of the shell, which provides the user with either a text-based user interface, commonly referred to as a terminal or a console, or a graphical user interface (GUI). Almost all personal computers and mobile devices have graphical shells, but the vast majority of servers only have terminals. These shells receive commands from the user, and they provide user applications when requested.

GNU/Linux

BSD is both kernel and userland in one. Linux, on the other hand, is merely the kernel. Many of the userland tools come from another project. Before Linus Torvalds began developing Linux, Richard Stallmen attempted to provide a free UNIX-compatible operating system call GNU OS. The GNU project was not successful in developing a kernel, but it created many userland tools. The resulting Linux operating system is a combination of the Linux kernel and GNU userland. Because the userland tools come from the GNU project, Richard Stallman insists that the Linux operating system should actually be called GNU/Linux to highlight the contributions of both projects. The Debian GNU/Linux distribution has adopted this name, but most others prefer the shorter label of just Linux.

Command Line

Linux graphical shells, also called desktop environments or window managers, function similar to the graphical shells of other desktop operating systems. Much of the power of Linux is in the text shell, and system administrators almost exclusively use the command line in a terminal to manage Linux computers. Knowledge with Linux terminal commands is a valuable competency for an accounting architect. The most basic commands are the same across all Linux distributions, as well as BSD and UNIX. As a result, familiarity with a few line commands can immediately unlock basic functionality of most enterprise-grade operating systems. More advanced commands, such as how to control system daemons or the package manager, vary from distribution to distribution.

Package Managers

One of the trademarks of BSD and Linux is the package manager. The package manager manages the installation and updating of all software in the operating system, including the operating system kernel. The benefit of a package manager is two-fold. First, the package manager provides the user with access to a vast collection of software repositories (i.e., data stores) that contain most applications the user would need. Linux and BSD users rarely need to search the Web for applications to download and install because the software repos often have a solution. Second, the package manager provides a single solution for keeping all applications and the operating system itself up to date.

Linux Distributions

Although over 600 different Linux distributions currently exist, many of these vary only by the software applications included by default with the operating system. However, a user can have distro A mimic distro B simply by using the package manager to install the applications from distro B on distro A.

A much smaller number of distributions families exist. These families differ from one another by the choice of package manager. The package manager determines software compatibility, and an application written for one package manager will not be compatible with an operating system that uses a different package manager. The most common package managers are DEB-based; the second most common are RPM-based.

Dependencies

Software applications rarely include all code necessary perform their intended functions. One reason for this is that multiple applications can share the same functionality and duplicating the code would be inefficient. As a result, many applications rely on software libraries. Just as a physical library provides a shared store of knowledge, a software library provides applications with a shared store of functionality.

Libraries that an application needs in order to function properly represent dependencies of that application. Many package managers identify and install all dependencies when installing a software application. This makes software administration much easier for the user.

Firmware Interface & Bootloader

The kernel and userland make an operating system, but in order for the operating system to be available to the user, a firmware interface is necessary. Firmware is software embedded hardware that controls that hardware. Two common computer firmware interfaces are Basic Input/Output System (BIOS) or, more recently Unified Extensible Firmware Interface (UEFI). This firmware resides on the motherboard's own flash storage, and it performs two important functions. The first is to initialize and test the computer hardware, and the second is to load the bootloader. The bootloader contains the instructions necessary to locate and boot the operating system kernel. Once the bootloader initializes the kernel, the operating system kernel launches any necessary services, daemons, and the user shell.

Putting It All Together

Powering the Software

The operating system is a complex collection of kernel and userland tools. Designers and maintainers of information systems must understand operating system administration and functionality, but not for the sake of the operating system itself. The purpose of the operating system is to power the information system applications, provide data storage, and serve and process data for the users.

Most companies have chosen Linux as the operating system of choice for their information system. The Linux kernel is one of the most successful open source projects. More than 13,000 developers contribute more than 4,500 lines of code per day. It powers the Android operating system and the Web services of Netflix, Amazon, Google, and many others.
