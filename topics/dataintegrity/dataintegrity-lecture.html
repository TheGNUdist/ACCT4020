Systems Integrity

Accounting Architecture

The final two blocks of the Control section are Availability and Processing Integrity. These terms correspond to matching concepts in the Trust Services Framework. Combined, these blocks prevent data loss and data errors.

Availability

The system is available for operation and use as committed or agreed.

Two threats of availability are data loss and downtime. Three tools combat these threats:

  • Interpretation preserves accessibility of legacy data.
  • Redundancy avoids data loss and unplanned downtime.
  • Change management limits planned downtime.

Interpretation

I cannot say what the original reason was for the Unix philosophy that the basis of all data is plan text, but a significant benefit of that mindset is a maximization of data accessibility. Binary files require specific hardware and software to access, so users have two choices (1) retain in perpetuity the hardware and software necessary to access the data or (2) emulate the hardware and software.

Emulation

Similar to virtualization, emulation uses software to mimic hardware. However, in the case of emulation, the hardware is non-existent, and software translates hardware requests into the proper architecture and communicates with the actual hardware. (Incidentally, this is a popular practice for playing old console and arcade games on personal computers.)

Redundancy

System components can fail. A single point of failure is a failure that stops an entire system, and redundancy is designed to avoid this phenomenon. A duplicate site creates redundancy for an entire cloud. A server cluster, in addition to increasing availability by adding computing power to support multiple simultaneous client requests, allows alternative servers to take over when one server fails. Storage backups create duplicates of data in case storage devices become corrupted.

RAID

The lowest level of logical volume that we will consider is the Redundant Array of Independent Disks (RAID). A RAID combines multiple physical disks or volumes into a group of logical volumes. RAID can take on many forms, and each form has a number. RAID 0 involves combining two physical volumes into one logical volume, called a stripe, so that the file system only sees a single drive with the storage capacity of both physical volumes combined.

Full Backups

Full backups create a copy of all current data. The benefit of full backups are that only one restoration is necessary to bring the system back to the most recent backup. The drawback is that full backups are large. They store everything, which means that each object that existed prior to a backup is stored in every subsequent backup, as well.

Incremental Backups

Incremental backups only store the changes (deltas) since the previous backup. This is the backup method used by Git and other version control systems. In order to restore an entire backup, it is necessary to restore every delta. This process can become impossibly long. As a result, many backup solutions combine these two methods by having frequent incremental backups and infrequent full backups.

Denied!

One major threat to system availability is denial of service (DoS). This involves sending a small network package (i.e., ping) to a server and asking for a response. If the number of pings becomes too large, this otherwise innocuous communication can overwhelm resources and effectively shutdown a server. Frequently, the attack is distributed across multiple clients in a distributed denial of service (DDoS) attack.

Processing Integrity

System processing is complete, valid, accurate, timely and authorized.

The goal of every information system is to convert raw data into useful information by avoiding errors and corruption:

  • Data validation minimizes data entry and processing errors.
  • Transaction processing maintains integrity of database records.
  • Refreshing ensures that analyzed data is up-to-date.

Cryptographic Hash

Even though cipher text is not human- or computer-readable, it can still provide useful information even when decryption is not possible. In fact, some encryption keys (e.g., MD5) are designed to create a cipher text that cannot be decrypted. The purpose in this case is not to protect a message in transit but rather to verify the integrity of the data. Verification has two components. The first involves confirming that the data has not been modified, and the second involves confirming the identify of the sender.

Data Entry

Human intervention can always result in errors. Data validation minimizes the possibility of data entry errors in multiple forms:

  • Data type verifies that content has the correct data type.
  • Check digits determine that a string of digits follows allowable algorithms (e.g., credit cards, ISBN).
  • Presence check looks for missing values in a record.
  • Range check confirms that a value is between a predetermined minimum and maximum.
  • Uniqueness check ensures that unique values are not duplicated (e.g., usernames).
  • Look-up check limits selection to pre-existing values (e.g., approved vendors).

Data Processing

Data validation also minimizes errors in data processing. A common example is the batch or hash total, which looks for missing records when processing a group (or batch) of records by comparing a value for the group to the aggregation of individual values (e.g., total hours worked compared with the sum of hours worked in each record).

ACID

One of the biggest benefits of RDBMS and drawbacks of NoSQL is transaction processing, which prevents a transaction from only being partially completed. In order for a database to be transactional, it must satisfy ACID:

  • Atomicity: all changes happen or none happen.
  • Consistency: all actions as a group violate no constraints.
  • Isolation: each transaction is treated as occurring before or after, not during, any others.
  • Durability: each committed transaction's changes are permanent.

Refreshing

In a data warehousing environment, the transaction systems contain one version of the data, and the data warehouse and data marts contain another version. Accurate data analytics relies on current data in the warehouse, which requires periodic refreshing to bring the warehouse copy up to date.

Redundancies

The benefits of a database system rely on the avoidance of redundancy. In fact, one of the reasons to transition from a file system to a database system is to avoid redundancies from separate individuals tracking information in separate flat files. However, redundancies can also arise in database systems through the creation of functional dependencies. These dependencies create problematic create, update, and delete anomalies.

Partial Dependencies

A partial dependency arises if a non-key attribute (i.e., an attribute that is not a primary key) depends on only part of a composite key. Again using the example of the course list, the composite key Semester, Year, Course, and Section identifies a unique record. However, the attribute Course Name exclusively depends on Course, which is only part of the composite key.

Transitive Dependencies

A transitive dependency arises when a non-key attribute is dependent on another non-key attribute. For example, the entity Employee might include Employee ID, Employee Name, Department ID and Department Name. Employee ID is the primary key, and the employee's department (Department ID) depends on Employee ID. However, Department Name depends on Department ID, which is not a primary key. This is a transitive dependency because Employee ID → Department ID → Department Name, which creates a transitive link between Employee ID and Department Name.

Create Anomaly

The create anomaly arises when it is not possible to add a record to a table because the new record has no value for the primary key. For example, suppose the Inventory table with SKU as primary key includes the name, address, and contact info for the vendor. The company wishes to add an approved vendor to the vendor list, but the company has not yet identified with items to order from that vendor. It is not possible to add that vendor to the database because no value for the primary key SKU exists.

Normalization

Normalization removes functional dependencies and their resulting anomalies by dividing tables with dependencies into multiple tables connected by foreign keys. Second normal form (2NF) removes partial dependencies, and third normal form (3NF) removes transitive dependencies. First normal form (1NF) also resolves anomalies, but it is too simplistic for our purposes. Third normal form is the minimum standard for a well-defined database.

Many-to-Many

One indication that a dependency exists is a relationship between two tables with maximum cardinalities of “many” in both directions. The resolution of this issue is a linking table that includes the primary keys of both original entities as a composite primary key. Because these are also the primary keys of another entity, they are simultaneously primary and foreign keys.

Normalizing an E-R diagram

This E-R diagram entity lists common attributes for a sales order. (By the way, extension is unit price multiplied by quantity.) However, it is not 3NF. Construct an E-R diagram that includes each of these attributes—except those that meet the criteria in the “Exceptions to Entities and Attributes” slide—in a set of 3NF entities.

Customer

First, we create a Customer entity. By storing this data in a separate table, we avoid duplicating it each time that the customer places a new order. We select the customer's email address as a primary key [PK] because customer accounts are frequently identified by the email address.

Carrier

Second, we create a Carrier entity to avoid duplicating carrier information for each order. The Carrier ID primary key could simply enumerate the carriers (i.e., 001, 002, 003, etc.). Not all primary keys have any interpretation. Sometimes they are nonsensical and only serve to uniquely identify records.

Inventory

Third, we create an Inventory entity. Although quantity on hand is not normally on a sales order, it is an important Inventory attribute, so we add it to the Inventory table. SKU is an inventory identifier that is common to vendor and customer, so we can use it as an informative primary key.

Sales Tax

Fourth, we create a Sales Tax entity to store the sales tax rates for each jurisdiction. Because jurisdictions are frequently identified by ZIP code, we can use that as the primary key.

Relationships

Next, we assign relationships to the entities and cardinalities to the relationships. This involves some assumptions based on hypothetical business rules. Each customer is associated with at least zero and at most many sales orders. Each carrier is associated with at least zero and at most many sales orders. Each sales order is associated with exactly one customer and one carrier.

Relationships

Although sales tax applies to a sales order, the customer's ZIP code resides in the Customer table. As a result, we will link the Sales Tax entity with the Customer entity with the assumption that each customer (by virtue of the customer's address) is associated with one sales tax rate, while each ZIP is associated with at least zero and at most many customers.

Relationships

Finally, each sales order is associated with at least one and at most many inventory items, and each inventory item can be associated with at least zero and at most many sales orders. This many-to-many relationship is a problem, which a linking table will fix.

Foreign Keys

Before creating a linking table, we add foreign keys to operationalize relationships. Customer email, carrier ID, and SKU are foreign keys in the Sales Order table, and ZIP, which already was an attribute of the Customer entity becomes a foreign key. This links customer ZIP to sales tax rate, and it restricts customers to ZIP codes already in the Sales Tax table.
