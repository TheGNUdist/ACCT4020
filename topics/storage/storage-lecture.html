Digital Storage

Accounting Architecture

Storage is a unique block in the Technology leg of the Architecture. It is not independent of the Hardware and Software blocks because digital storage comprises a combination of hardware and software components. Also, unlike the other blocks of the Architecture, the Storage block maps directly into a step of the Information Life Cycle: Storage.

Types of Storage

  • File systems contain data in separate files and folders. This is the traditional storage format we use in our individual computers.
  • Database systems store all data in a single location.

These two systems serve can accommodate physical and digital resources. However, for our purposes, we will exclusively focus on digital storage systems.

How Storage Looks

Because storage is a combination of hardware and software, storage contents can appear differently under different circumstances. These different views or schemata (plural of schema) are not different storage methods, but rather distinct representations of the same data.

  • Physical view: how the data is stored on physical disks. For the most part, this view is too technical for our purposes.
  • Logical view: how the file or database system presents the data. This view applies to system administrators.
  • User view: how the end-user application presents the data. This view applies to end users.

A Simple Example

A single audio file on a personal computer represents a simple example of the three storage views. The audio file is broken into small blocks of data stored on the physical drive; the operating systems file system shows those blocks as a single logical file within a hierarchical system of folders (or directories, for those from the old school); a music player application uses the data in the file to send music to the user via the audio daemon and the metadata to display information about the track, such as name, artist, album, year, etc.

A Logical Rabbit Hole

It is relatively straightforward to understand that the same data can have multiple different user views. Depending on compliance and business need, different end users will see different contents. For example, students can see their own grades, but professors can see the grades of every student. Similarly, multiple logical views of data exist, but their characteristics and distinctions are more technical and more difficult to comprehend.

Volumes

The lowest level of storage is the physical volume. For illustrative purposes, we can think of the physical volume as a partition, which is a portion of a physical disk. One or more logical volumes may sit between the physical volume and the file system, but they are not necessary. However, in the current technological landscape, at least one logical level between the physical volume and the file system is common.

RAID

The lowest level of logical volume that we will consider is the Redundant Array of Independent Disks (RAID). A RAID combines multiple physical disks or volumes into group of logical volumes. RAID can take on many forms, and each form has a number. RAID 0 involves combining two physical volumes into one logical volume, called a stripe, so that the file system only sees a single drive with the storage capacity of both physical volumes combined.

Hardware-level RAID Controllers

Three different devices can create a RAID. A hardware RAID uses specialized hardware to control the RAID and the underlying disks. This is the most sophisticated, most expensive, and rarest form of RAID. The next level down is the motherboard RAID. In this scenario, the motherboard acts as RAID hardware and controls the RAID. Because this only simulates a hardware RAID, users frequently refer to this as fake RAID, and many individuals feel that this form of RAID is too unreliable to be a safe option.

Software-level RAID Controllers

In two prior cases, any operating system on the hardware will only see the logical volume created by the RAID and not the physical volumes. The simplest form of RAID, as well as the most popular, is the software RAID. Here, the operating system sees the physical volumes and creates and manages the logical RAID volume. This is a popular RAID solution for workstations because it is cheaper than a hardware RAID and more reliable than a fake RAID, but it has one problem. The computer can only have one operating system at a time (i.e., dual booting is not possible).

Logical Volume Management

A higher level of logical storage is Logical Volume Management (LVM). LVM may sit directly on top of physical volumes, or it may sit on top of a RAID. LVM creates a volume group which can comprise as little as one physical partition or as much as multiple physical volumes. LVM then creates one logical volume within the volume group for each file system. One of the benefits of LVM is the ability to resize the logical volumes and their file systems at will, whereas partitions have a preselected static size.

File System

The file system provides the familiar logical hierarchical folder structure. Different operating systems support file systems (NTFS for Windows, HFS+ for OS X, EXT4 for Linux, ZFS for BSD). The differences between these is too technical for our purposes, but many consider ZFS to be the most awesome. The operating system and other application software reside on the file system.

Database System

The logical view of a database system is a centralized data store, but the data in the database actually resides on a file system. The database server collates the data in the file system and presents in an centralized manner. The most widely implemented database systems are either relational database management systems (RDBMS) or non-relational databases (NoSQL).

Each relational database contains one of more two-dimensional tables. The technical name for a table is a relation, hence relational database. These tables are similar to spreadsheets found in office software. The individual rows in the relations are called records or tuples. The individual columns are called attributes or fields. The intersection of a row and a column is a cell. Cells hold a single value.

Non-relational Databases

Generically speaking, any database that is not relational could be considered non-relational, but in practice, non-relational (Not Only SQL; NoSQL) databases are a specific class of databases that abandoned the principles of relational databases in favor of alternative models. Although this class comprises several distinct models, we will focus on three popular examples: key-value stores, document stores, and column stores.

Key-value Store

Key-value is the model that governs all metadata. A key-value store is a database that pairs unique keys or attributes (i.e., unique identifiers) with data objects. The data object can be anything from a primitive to an entire file (e.g., document, audio file, etc.). The role of the key-value store is to facilitate efficient storing and retrieval of values (i.e., data objects) by their keys.

Document Store

A document store is a special case of a key-value store in which the value is a text file with a multiple metadata attributes and values, frequently in XML format. The document store is essentially a condensed version of a relational database, except that instead of storing the individual values as fields in a record of a relational table, the document store keeps all values together in a single field (the entire document).

Column Store

A column store is likely also a special case of a key-value store, but it may not be very helpful to think about it in that manner. Each column contains a key and a value in the same sense as a metadata attribute and value pair. Additionally, each column has a timestamp. Wide-column stores can combine multiple related columns into rows that approximate a single record in a relational table except that because the attributes and values are paired in each column, each row can have a different selection of attributes.

Benefits of NoSQL Databases

  • Scalability: Because of the simplicity of each record in the database, NoSQL databases are designed to store many more records than their relational counterparts.
  • Speed: Relational databases require data entry transactions to find the correct table and attribute for each value; NoSQL databases take the data input and simply append it to the end of the store. For example, a relational database might need to find customer name, address, date, inventory number and unit price fields in order to store a new sales order. A column store could simply create a new column for each attribute and combine them into a row. A document store could take the order and place all attributes and values into a single document.

SQL

Structured Query Language is the language for interacting with relational databases. Although the name NoSQL literally stands for "Not Only SQL," many NoSQL databases have no support for SQL, and instead rely on other languages, such as Java, JavaScript and Python for data manipulation. One exception is Apache Cassandra, which can use either Java or CQL (Cassandra Query Language), which is a derivative of SQL. Because of the prevalence of relational databases and its ease of use, SQL remains a popular tool, and an increasing number of Big Data and data analytics systems have added support for this language.

Two Types of Models

  • Activity models describe a series of processes that comprise a system. This system may be a computer program or a business cycle involving many programs and people.
  • Structure models describe the characteristics of people, assets and data and their relationships with one another.

Previously we used activity models to diagram business processes. In this lecture, we will investigate structure diagrams (or data structure diagrams) as a tool for modeling data and databases.

Why More Diagrams?

Once we know how to diagram business activities, why is it necessary to create another diagram about the objects in those activities?

A process diagram identifies players (agents, processes and data objects) in a system, but an information system needs more dimensionality for two reasons. First, without characteristics, each activity would have duplicate objects because no traits would distinguish each instance of an object. Second, employees cannot interact with the black box that a trait-less object would represent. For example, no one can fill a customer order if the only available information is that a customer order exists.

Characteristics and Metadata

The Creation block of the Information section of the architecture addresses data classification using metadata. These metadata are the characteristics of the objects in the structure diagram. Although data storage is a maintenance activity and classification a creation activity, the fact that the data storage must store the metadata is further evidence of the need for a completely designed information system prior to data acquisition. Because structure diagrams include object characteristics, these diagrams help to identify necessary metadata.

Structure Models for Relational Databases

Despite the benefits of non-relational databases—we will discuss an additional benefit when we cover data manipulation later in the course—we will limit our discussion of structure models to relational databases. Three common structure models exist:

  • Entity-Relationship (ER; E-R; ERD) diagrams
  • UML class diagrams
  • Resource-Entity-Agent (REA) diagrams

Entity-Relationship Diagrams

The entity-relationship model initially introduced by Peter Chen in 1976 is the de facto standard for modeling relational databases. As the name indicates, the components of an E-R diagram are entities and relationships between entities. Additionally, thorough diagrams also include a series of attributes for each entity and cardinalities of the relationships.

Unfortunate Confusion

E-R diagrams have three data models: conceptual, logical and physical. Unfortunately, these fail to correspond directly to views in the Three Schema Concept. Both the conceptual and the logical model relate to the conceptual schema, and the physical model relates to the logical view. Furthermore, interpretation of these three terms is not consistent across user. As a result, we will restrict our terminology to the four views we have previously identified.

Entities

An entity is something of importance that needs to be represented in the database. Examples of an entity may be Employee, Invoice, Inventory, Time Stamp, etc. It is important to note that these entities are concepts and not manifestations of concepts. For example, Employee is an entity, but a particular employee, Joe Smith, is an instance of that entity. Entities correspond to tables in the database, whereas instances correspond to records in those tables.

Relationships

A relationship is a link between entities. Because entities are nouns, relationships can be described as verbs of one entity acting on another. Examples of this would be the relationship of Employee and Purchase Requisition. These two entities have a relationship describable as: Employee Creates Purchase Requisition.

Attributes

An attribute is any relevant characteristic of an entity. These correspond to the columns in a database table. Common examples might include Name, SSN#, SKU, Customer ZIP, etc.

Cardinalities

The magnitude of the potential relationships between an instance of one entity and the instances of another (i.e., between a record of one table and the records of another table) is cardinality. Cardinalities have three potential values: zero, one and many. Cardinalities also have two bounds: minimum and maximum. For example, if the two entities are Customer and Sale, then a customer is associated with at least (minimum) zero and at most (maximum) many sales, and each sale is associated with at least one customer and at most one customer. When the minimum and maximum cardinalities are equal, then it is only necessary to report one: each sale is associated with exactly one customer.

Coyne Rule of Zero

Although this is not a widely accepted rule of cardinalities, I believe that each relationship must have at least one minimum cardinality of zero. The reason for this is that the database can only create one record at a time. If a sale is associated with at least one customer, then the customer record must exist prior to the sale. If however, a customer is associated with at least one sale, as well, then the sale record must exist prior to the creation of the customer. However, they cannot both exist first. As a result, one of the minimum cardinalities must be zero.

Exceptions to Entities and Attributes

Not everything that is important to represent is an entity, and not every relevant characteristic is an attribute. The exceptions to this rule are potential entities and attributes that are combinations of pre-existing attributes. One simple example of this relates to Inventory costs. Unit Cost and Quantity on Hand would be an attribute of the entity Inventory, but Total Cost (= Unit Cost x Quantity on Hand) would not. Another example would be employee supervisors. Supervisors would not be an entity because it would simply be a subset of the Employee entity. These would-be entities and attributes become user views because they are not stored in the database but can be provided to the user.

Keys

A special subset of attributes that uniquely identifies a specific instance of an entity is a primary key. Each value of a primary key can only occur once per entity. Because these primary keys uniquely identify an instance, they also serve to link entities together to implement relationships. A primary key from one entity is included in another entity as a foreign key to create a link. For example, Employee might include Employee ID as a primary key, and Sales might include Employee ID as a foreign to identify the employee responsible for the sale.

Composite Keys

Sometimes no single attribute uniquely identifies an instance of an entity. For example, neither Semester nor Year nor Course nor Section could be a primary key in a course schedule, but the combination of Semester and Year and Course and Section can. Primary keys that comprise multiple attributes are composite keys.

Redundancies

The benefits of a database system rely on the avoidance of redundancy. In fact, one of the reasons to transition from a file system to a database system is to avoid redundancies from separate individuals tracking information in separate flat files. However, redundancies can also arise in database systems through the creation of functional dependencies. These dependencies create problematic anomalies.

Partial Dependencies

A partial dependency arises if a non-key attribute depends on only part of a composite key. Again using the example of the course list, the composite key Semester, Year, Course and Section identify a unique record. However, the attribute Course Name exclusively depends on Course, which is only part of the composite key.

Transitive Dependencies

A transitive dependency arises when a non-key attribute (i.e., an attribute that is not a primary key) is dependent on another non-key attribute. For example, the entity Employee might include Employee ID, Employee Name, Department ID and Department Name. Employee ID is the primary key, and the employee's department (Department ID) depends on Employee ID. In other words, the instance of department depends on which employee is referenced. However, Department Name depends on Department ID, which is not a primary key. This is a transitive dependency because Employee ID → Department ID → Department Name, which creates a transitive link between Employee ID and Department Name.

The create anomaly arises when it is not possible to add a record to a table because the new record has no value for the primary key. It is not possible to add that vendor to the database because no value for the primary key SKU exists.

Normalization

Normalization removes functional dependencies and their resulting anomalies by dividing non-normalized tables into multiple tables connected by foreign keys. Second normal form (2NF) removes partial dependencies, and third normal form (3NF) removes transitive dependencies. First normal form (1NF) also resolves anomalies, but it is too simplistic for our purposes. Third normal form is the minimum standard for a well-defined database.

Many-to-Many

One indication that a dependency exists is a relationship with maximum cardinalities of many in both directions. The resolution of this issue is a linking table that includes the primary keys of both original entities as a composite primary key. Because these are also the primary keys of another entity, they are simultaneously primary and foreign keys.

Indexing

One of the steps in the Maintenance section is indexing. This involves the creation of list of keys and pointers to increase the findability of those keys in a collection. One simple example of this is the index in the back of a textbook. The keywords are the keys, and the page numbers on which those keywords appear are the pointers. Database indices serve the same purpose. A single relation may have many records, and the DBMS may need a large number of processing cycles to find the records that a user wants. An index shrinks the size of the relation to a particular attribute and identifies which records in the relation have a particular key. This increases the speed of record retrieval.
